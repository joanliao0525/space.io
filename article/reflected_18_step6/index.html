第一次自己從頭生成一個專案，犯的錯誤實在是自己都看不下去，

所以就來寫個匯總吧😂


助教大人🐤提點：

「status 是 task 的一個欄位不是 table ，不應該出現在 table 的關聯設定裡。
建議閱讀：Active Record 關聯」

菜鳥🤦🏻‍♀️心得：

在建立 model 時，欄位屬性鍵入 references，會自動產生關聯，所以再決定欄位屬性時要注意，若該欄位不是外部鍵，則無需使用 references 使用 integer 即可。


助教大人🐤提點：

User 有很多的 Task ? 似乎漏掉關聯設定

菜鳥🤦🏻‍♀️心得：

粗心沒有檢查到，自己寫的 code 要檢查再檢查。


助教大人🐤提點：

view 不應該有 Task.new ，MVC 應該是將 controller 的變數 @task 放到 View
建議閱讀：Model、View、Controller 三分天下

菜鳥🤦🏻‍♀️心得：

原本是為了功能正常顯示，但是這樣的做法是錯誤的，不只要考慮到可讀性，
還要考慮到維護的方便性以及 code 本身的慣例及用法，還需研讀更多文件。


助教大人🐤提點：

這裡的縮排看起來是四格空白，rails 慣例是兩個空白

菜鳥🤦🏻‍♀️心得：

可以直接在 VSCode 或其他程式碼編輯器中設定 tab 縮排的預設空格數，
且在寫 code 的過程中注意縮排及格式，遵照 Ruby on Rails 慣例走。


助教大人🐤提點：

建議這下面加一行空白增加可讀性

菜鳥🤦🏻‍♀️心得：

隨時檢查檔案內容是否遵照 Ruby on Rails 慣例，並且注意可讀性、舒適性。


助教大人🐤提點：

permit(:title,:description) 這邊習慣上會寫這樣，
permit(:title, :description) 中間會多一個空格增加可讀性

菜鳥🤦🏻‍♀️心得：

同樣隨時檢查檔案內容是否遵照 Ruby on Rails 慣例，並且注意可讀性、舒適性。


助教大人🐤提點：

注意縮排，但是有個問題，為什麼一直出現沒看過的新增，理論上一支 pr 除非被要求更改，
不然不會一直出現沒看過的新增檔案，還是說這些修改應該是下一隻 pr 的？ＱＱ

  • 讓應該在下一隻 pr 裡的變更不要出現在這個 pr 裡，
  • git reset 到想要的節點
  • git stash -u 可以暫存變更(-u 包含 untracked files)
  • git stash pop 或 git stash apply 可以拿出暫存(pop 拿出來之後 stash list 就不會有了)
  • 假如是想放在其他分支上以後也可以用 git cherry-pick 撿到想要的分支上。
    建議閱讀：Git 面試題

菜鳥🤦🏻‍♀️心得：

要多練習 git 指令，不要浪費好用的工具。

助教大人🐤提點：

修改好的話可以點 resolve conversation，這樣就可以知道哪些已經修改好了哪些還沒。

菜鳥🤦🏻‍♀️心得：

有空可以多多使用 GitHub 發掘功能，不要浪費好用的工具。
